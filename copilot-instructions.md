# Rune AI Web PPT 项目开发指南 & 上下文记忆 (Instructions)

这个文件记录了本项目的核心架构、设计规范、开发流程以及当前的进度。在后续的会话中，请务必先读取此文件以恢复记忆，确保开发连贯性。

## 0. 强制预读规则 (Must Read Before Any New Slide)
- **硬性要求**：后续每次开始生成、修改或重做任何新页面前，必须先完整阅读一遍本文件，再开始设计与编码，**不得凭记忆直接开工**。
- **最少执行动作**：
  - 先读完本文件中的 `全局 UI / UX 规范`、`严格的开发工作流`、`前 11 页沉淀的设计母版`、`当前项目进度`
  - 确认本次新增页面要延续哪一种已验证布局母版，再决定 HTML 结构和 CSS 方向
  - 若发现新需求与既有母版冲突，必须先说明冲突点，再调整
- **默认原则**：优先复用前 11 页已经验证过的组件语言、色彩秩序、动画节奏和信息组织方式，而不是临时发明一套新风格。

## 1. 项目概述与技术栈
- **目标**：为“Rune AI（企业级智算平台）”制作一个企业级、高交互体验的 Web 版幻灯片（PPT）。
- **技术栈**：纯原生前端（Vanilla HTML5 + CSS3 + JavaScript），**不使用任何第三方框架**（无 React/Vue/Tailwind 等）。
- **视口逻辑**：固定 1920x1080 (16:9) 设计稿比例，通过 `app.js` 监听窗口大小并利用 `transform: scale` 动态等比缩放整个 `#viewport`，确保在任何屏幕下都100%无缝展示。

## 2. 全局 UI / UX 规范
- **导航控制**：采用“边缘触发（Edge-hover）”模式。默认全屏沉浸，不显示任何 UI。只有当鼠标移入屏幕左右边缘或底部区域（热区）时，才会浮现翻页按钮和底部进度条。
- **翻页交互**：除按钮、键盘与触控滑动外，默认也要支持鼠标滚轮/触控板上下滚动翻页；页面不应出现原生纵向滚动条。
- **排版字体**：极致现代科技风。英文字体优先使用 `Inter` (Google Fonts)，中文优先 `PingFang SC`。
- **全局水印**：以 **Slide 02 右上角** 的表现为标准，在非封面/尾页的右上角默认全局显示“Logo 图 + 晓石云”的水印。该水印必须固定锚定在整个 `#viewport` 的右上角，不能跟随单页内容区或受到单页 `padding` 影响，并且必须放在独立 overlay 层中作为全局置顶元素。后续新增页面默认都必须保留这套全局水印；如需隐藏，只有在用户明确要求时，才可在对应的 `<section class="slide">` 上添加 `.no-watermark` 类名。
- **文案规范**：全项目静止使用 emoji，所有标题、标签、说明文案统一采用纯文本表达。
- **CSS 架构**：
  - `global.css`: 定义品牌色（主蓝 `#1E60F8`、副绿、辅紫）、字体和阴影。
  - `slide-framework.css`: 核心缩放容器和全局水印样式。
  - `navigation.css`: 隐藏式交互 UI 样式。
  - `animations.css`: 全局过渡与动效管理。
  - `slides/XXX.css`: 每个 Slide 单独对应一个 CSS 文件，避免样式污染。
- **页面定位机制**：所有 `<section class="slide">` 必须保持真正的幻灯片叠放逻辑，依赖 `slide-framework.css` 中的绝对定位显示/隐藏；单页 CSS 不得再把整页容器改回普通文档流定位，避免 hash 路由退化为锚点滚动。

## 2.5 前 11 页沉淀的设计母版 (Design System from Slides 1-11)

### A. 总体视觉原理
- **主基调**：白底、浅灰、浅蓝网格、柔光渐变、轻阴影的企业科技风；避免重黑大面积压迫感。
- **页面气质**：专业、克制、产品化，不做炫技式视觉堆叠；重点是“结构清晰 + 信息高密度 + 投屏可读”。
- **品牌色秩序**：
  - 蓝色：平台、基础设施、标准化能力、主链路
  - 绿色：协作、数据、社区、稳定运维
  - 紫色：模型交互、生态扩展、生成式能力
  - 琥珀色：商业化、市场、模板、业务价值
- **背景语言**：
  - 优先使用 `radial-gradient + linear-gradient + 轻网格` 组成的白底科技画布
  - 可按主题加入轨道圆环、中心光晕、轻量同心圆，但必须弱化到背景层，不能喧宾夺主
- **组件材质**：
  - 主卡片：白底 / 浅渐变 / 细边框 / 柔阴影
  - 总结卡、亮点卡：更平、更轻，作为说明层
  - 终端或工程模块：允许使用深底，但只作为局部强调，不可整页转暗

### B. 信息组织原则
- **每页必须先有一句核心观点**：标题和副标题必须让观众在 3 秒内理解该页想表达什么。
- **摘要条已成为默认组件**：从中段页面开始，优先使用 4 到 5 个轻量标签先建立理解框架，再进入主体内容。
- **内容不做散点堆砌**：每页都必须有明确的信息骨架，例如“双栏”、“三段式”、“中心发散”、“卡片墙”、“总框分层”。
- **底部结论条高频使用**：当页面信息较多时，底部用一句高对比总结条收束价值，形成章节记忆点。
- **投屏优先**：宁可减少说明层，不要让整页在 16:9 中出现密集小字、重叠或靠边拥挤。

### C. 版式母版库
- **母版 1：封面中心式**
  - 代表页：Slide 01
  - 特征：大标题、弱背景纹理、底部信息、品牌主视觉
- **母版 2：左右双栏 + 底部结论**
  - 代表页：Slide 02、Slide 05、Slide 10、Slide 11
  - 特征：左侧主视觉/结构图/矩阵，右侧说明或控制台，底部结论带
- **母版 3：中心舞台 / 中轴核心**
  - 代表页：Slide 03、Slide 09
  - 特征：中心核心节点最重，外围能力模块围绕展开，适合讲“平台中枢”或“网关枢纽”
- **母版 4：单画布分层总览**
  - 代表页：Slide 04
  - 特征：一个总框承载多层结构，自上而下或自下而上表达平台全景
- **母版 5：卡片墙**
  - 代表页：Slide 06
  - 特征：4x2 或规则矩阵，适合一次性承载多个并列能力点
- **母版 6：三段式流程 / 能力链路**
  - 代表页：Slide 07、Slide 08
  - 特征：从左到右或从上到下表达“接入 -> 处理 -> 输出”的业务链路

### D. 卡片与组件风格约束
- **标题区**：
  - 标题统一左对齐
  - 常配短下划线或标题后高亮短条
  - 副标题控制在 1 到 2 行，承担“核心观点”职责
- **摘要标签条**：
  - 使用圆角胶囊
  - 背景轻、边框细、字重高
  - 数量建议 4 到 5 个，避免超过 6 个
- **功能卡 / 价值卡**：
  - 默认使用圆角白卡 + 轻阴影
  - `hover` 只做轻微上浮和阴影增强，不做夸张位移
  - 编号、英文副标题、中文主标题是高频可复用语言
- **右下角说明卡语言**：
  - 当前以 Slide 10 / Slide 11 的亮点卡为基准：顶部短色条、蓝色标题、深灰正文、轻量 hover
- **结论条**：
  - 深蓝或蓝黑渐变背景
  - 白字，字重偏高
  - 作用是收束，不承载复杂信息

### E. 动效与交互节奏
- **统一节奏**：
  - `0.12s` 左右：标题区先入场
  - `0.26s` 左右：摘要条进入
  - `0.38s - 0.60s`：主体分区或主卡进入
  - `0.82s` 左右：底部结论条收尾
- **动效类型**：
  - 以淡入、轻微位移、轻缩放为主
  - 大部分卡片使用 `translateY`、`translateX`、`scale`
  - 持续动画只保留在背景网格漂移、轨道旋转、流动光点等弱感知层
- **严禁事项**：
  - 不要让多个区域同时做重动画
  - 不要做抢主信息的复杂持续运动

### F. 前 11 页的具体风格结论
- **Slide 01**：纯白封面，左上 Logo，占大标题和底部信息，适合作为“品牌建立页”母版。
- **Slide 02**：左侧数据、右侧洞察与痛点、底部结论，是“问题定义页”标准解法。
- **Slide 03**：中心发散式平台使命页，适合讲核心平台与三大子系统关系。
- **Slide 04**：单画布四层总览，适合讲架构全景。
- **Slide 05**：延续 Slide 02 的双栏容器逻辑，适合讲角色分工或两套视角对照。
- **Slide 06**：4x2 卡片墙，适合承接多个并列功能模块。
- **Slide 07**：三段式流程，适合讲“统一治理链路”。
- **Slide 08**：左能力列 + 右展示区，适合讲产品能力和截图组合页。
- **Slide 09**：中轴网关编排页，适合讲请求汇聚、路由和治理。
- **Slide 10**：左资产矩阵 + 右控制台/终端/亮点卡，适合讲资产社区或仓库型产品。
- **Slide 11**：左应用市场主界面 + 右部署/运维价值卡；当前版本已去掉模板带和差异化卡，仅保留四张右下角说明卡，并且其视觉语言已对齐 Slide 10 的亮点卡。

### G. 后续页面默认沿用规则
- 新页如果没有特别说明，优先从上述 6 种母版中选一种，不要重新发明版式。
- 新页如果属于“平台功能说明”，优先考虑 `双栏`、`卡片墙` 或 `三段式流程`。
- 新页如果属于“平台中枢/路由/架构关系”，优先考虑 `中心舞台` 或 `单画布分层总览`。
- 新页如果属于“真实产品界面展示”，优先做成“截图式控制台 + 说明卡”的产品化页面，而不是抽象海报页。

## 3. 严格的开发工作流 (Workflow)
请 AI 助手在今后的服务中严格遵守以下流程：
1. **先读 instructions 再开工**：开始任何新页面前，必须先重新阅读本文件，不得跳过。
2. **逐页推进**：采用**问答模式（一问一答）**。千万不要一次性输出或生成多个 Slide 的代码。必须在一页完整写完、用户审核确无误后，再推进下一页。
3. **独立记录 Skill**：每完成一页 Slide 的设计，必须将该页的排版思路、数据映射和特殊设计记录在 `skills/slide-XX.skill.md` 中留档。
4. **隔离开发**：新页面的 HTML 结构追加在 `index.html` 的 `#slides-wrapper` 中，新页面的 CSS 必须新建文件并通过 `<link>` 引入，不可写在行内。
5. **提交纪律**：从 Slide 04 开始，AI 助手每次完成一轮实际代码或说明文档变更后，必须立即执行一次 `git commit`，保持提交粒度清晰；待该页内容经用户最终确认后，再对该页相关提交执行一次 squash，整理为干净历史。
6. **视觉验收**：每次完成页面视觉或布局修改后，AI 助手都必须使用 Playwright 打开当前页面并截图核验，确认包括全局水印在内的关键元素实际可见、位置正确，再结束该轮修改。
7. **复用优先**：新增页面必须先判断它属于哪一种既有母版，再开始落代码；除非用户明确要求，否则不要偏离当前设计系统。
